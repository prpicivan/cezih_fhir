Poslovni kontekst: Kako teče pregled?

Integracija s CEZIH-om nije samo "slanje nalaza". Svaki pregled pacijenta prati strogi protokol razmjene podataka koji se sastoji od tri međusobno povezana objekta: Posjete, Slučaja i Dokumentacije. Razumijevanje ovog toka ključno je za ispravnu integraciju vaše G9 aplikacije.

1. Posjeta (Encounter)

Posjeta označava fizički dolazak pacijenta u ordinaciju. Svaki pregled, svaki kontakt = jedna posjeta. Bez otvorene posjete ne možete slati nalaz.

OTVORENA

→

ZATVORENA

TC14 — Pregled završen

📋 Poslovna pravila za Posjete:

  • • Posjeta se otvara prije pisanja nalaza (TC12 mora biti uspješan)
  • • Klasa posjete mora odgovarati tipu dokumenta: AMB za nalaz 011/012, IMP za otpusno pismo 013
  • • Posjeta se NE MOŽE zatvoriti dok postoje nepotpisani dokumenti vezani na nju
  • • Prije otvaranja nove, provjerite ima li pacijent već aktivnu posjetu (TC15) — možete se priključiti na nju
  • • Zatvorena posjeta se ne može ponovno otvoriti

2. Slučaj (Condition / EpisodeOfCare)

Slučaj predstavlja cjelokupni tijek liječenja jedne bolesti kroz više posjeta. Pacijent s hipertenzijom ima jedan slučaj koji traje godinama, a unutar njega mnogo posjeta i nalaza.

📋 Poslovna pravila za Slučajeve:

  • • ZLATNO PRAVILO: Ako pacijent već ima aktivan slučaj za istu dijagnozu (MKB-10 šifru), NE otvarajte novi — koristite postojeći cezihCaseId
  • • Jedan pacijent može imati više aktivnih slučajeva — ali svaki za drugu dijagnozu
  • • Svaki nalaz (TC18) mora biti vezan na jedan slučaj putem caseId
  • • Slučaj u remisiji ne znači da je zatvoren — bolest može biti u mirovanju, ali slučaj je i dalje aktivan u CEZIH-u
  • • Prije kreiranja novog slučaja (TC16) uvijek dohvatite postojeće (TC15)
  • • Dijagnoza slučaja je MKB-10 šifra — dohvatite je iz našeg šifrarnika (TC7/8)

Pregled sustava i Arhitektura

CEZIH FHIR Bridge je lokalni middleware (posrednik) dizajniran kako bi vašoj G9 aplikaciji omogućio brzu i bezbolnu integraciju s nacionalnim zdravstvenim sustavom (CEZIH). Umjesto višemjesečnog proučavanja IHE MHD profila, HL7 / FHIR STU3 standarda i složene PKI infrastrukture, vaša aplikacija komunicira s našim sustavom putem jednostavnih REST API poziva.

1

Vi šaljete JSON

Vaša aplikacija šalje obične, "plosnate" JSON objekte (npr. MBO pacijenta, tekst nalaza, MKB-10 šifru) na naš lokalni API.

2

Mi gradimo FHIR

Middleware u pozadini generira sekvencijalne OID-ove, pakira strogo validirani IHE MHD Bundle i vrši C14N XML kanonizaciju.

3

Potpis i Slanje

Pomoćne radnje potpisujemo automatski, a za dokumente vam izlažemo gotov UI Widget za unos PIN-a pametne kartice ili Certilie.

Osnovna pravila API komunikacije

  • Base URL Svi API pozivi usmjeravaju se prema: http://<adresa-servera>:3010
  • Content-Type Svi POST i PUT zahtjevi moraju sadržavati HTTP zaglavlje: Content-Type: application/json
  • Struktura Odgovora (Response Format) Svaki odgovor vraća JSON koji u rootu sadrži boolean vrijednost "success": true | false. U slučaju greške (success: false), detaljan opis razloga odbijanja nalazi se u polju "error".
  • 🔀 Podržani formati (Multi-Format Support) Naš Middleware je dizajniran da se prilagodi vašem sustavu, a ne obrnuto. Endpointi za slanje dokumenata izvorno podržavaju application/json. Međutim, ako vaš sustav već izvozi podatke u XML formatu (ili HL7 v2), javite nam se — omogućit ćemo vam namjenski XML/HL7 adapter unutar Middlewarea, tako da možete slati vaše postojeće formate bez prepisivanja koda vaše aplikacije!

2. Posjete (Encounters)

Dohvat aktivnih posjeta s CEZIH-a (TC15)

Prije otvaranja nove posjete provjerite ima li pacijent već aktivnu (in-progress) posjetu. Ako ima, možete nalaz vezati direktno na nju.

GET /api/visit/remote/{mbo}

Request Body

GET zahtjev. MBO je u URL-u.

Odgovor (Response)

{ "success": true, "visits": [ { "cezihVisitId": "cmmqr24hp037o5c85wrivv0z7", "status": "in-progress", "class": "AMB", "startTime": "2026-03-15T08:00:00Z", "practitionerName": "Dr. Ivan Prpić", "institutionName": "Dom zdravlja Zagreb" } ] }

Otvaranje posjete (TC12)

Označava da je pacijent ušao u ordinaciju i da liječenje počinje.

Pravilo klase (class):
Ambulantni/specijalistički nalaz (011, 012) → class = AMB
Otpusno pismo iz bolnice (013) → class = IMP
POST/api/visit/create

Request

{ "patientMbo": "999999423", "class": "AMB", "startDate": "2026-03-15T08:00:00Z" }

Odgovor

{ "success": true, "result": { "localVisitId": "local-uuid-1234", "cezihVisitId": "cmmqr24hp037o5c85wrivv0z7" } }

Zatvaranje posjete (TC14)

Kada je pregled gotov i svi nalazi su potpisani, posjetu morate zatvoriti. ID posjete prosljeđujete u URL-u.

POST /api/visit/{id}/close

Request

{ "patientMbo": "999999423", "endDate": "2026-03-15T08:30:00Z" }

Odgovor

{ "success": true }

3. Zdravstveni Slučajevi (Conditions)

Kako funkcioniraju Slučajevi?

Slučaj (Condition) predstavlja cjelokupni tijek liječenja jedne bolesti. CEZIH strogo prati "Stroj stanja" (State Machine) za svaki slučaj.

Zlatno pravilo:

Ako pacijent već ima otvoren slučaj za određenu dijagnozu, NE SMIJETE otvarati novi (TC16). Dohvatite postojeći (TC15) i proslijedite njegov cezihCaseId prilikom slanja nalaza.

Životni ciklus Slučaja:

↓ 2.3 Remisija ↓

↙ 2.5 Relaps

→ AKTIVAN

Pretraživanje nalaza (TC 21)

Dohvaća listu i metapodatke dokumenata na CEZIH-u za pacijenta. Middleware transparentno hendla CEZIH paginaciju podataka u pozadini.

GET/api/document/search-remote?patientMbo={mbo}

Odgovor (Response)

{ "success": true, "documents": [ { "id": "urn:oid:2.16...", "contentUrl": "https://doc-mhd.../?data=base64...", "typeDisplay": "Specijalistički nalaz", "date": "2026-03-14T10:00:00Z" } ] }

Dohvat i čitanje nalaza (TC 22)

Kada s prethodne rute dobijete contentUrl (ili OID) dokumenta, ovdje ga preuzimate. Middleware preuzima kompleksni FHIR Bundle s CEZIH-a i automatski mapira komplicirane FHIR resurse u vama poznati "plosnati" format (Anamneza, Status, Preporuka) spreman za ispis!

Vaše JSON Polje Originalna FHIR Lokacija (MHD)
anamnesisObservation (code=15)
findingObservation (code=16)
recommendationCarePlan > description
diagnosisCodeCondition > code
GET/api/document/retrieve?url={contentUrl}

Odgovor (Parsirani dokument)

{ "success": true, "document": { "anamnesis": "Pacijent se žali na bol u prsima...", "finding": "Fizikalni status uredan, tlak 120/80.", "recommendation": "Mirovanje i kontrola za 7 dana.", "diagnosisCode": "I10", "practitionerName": "Prpić Ivan" } }

Slanje nalaza (TC18)

Šaljete sadržaj nalaza s ID-jem posjete i slučaja. Middleware vraća documentOid za UI Widget.

📋 MKB-10 dijagnoza je obavezna!
Svaki klinički nalaz mora sadržavati šifru dijagnoze prema MKB-10 šifrarniku (polje diagnosisCode). Za brzu pretragu i autocomplete koristite naš endpoint: /api/terminology/diagnoses?q={upit} — vidi sekciju Pretraga dijagnoza.

Kako Middleware mapira vaše polje → FHIR resurs

Vi šaljete obični JSON. Middleware automatski gradi kompleksni FHIR IHE MHD Bundle koji CEZIH očekuje:

Vaše JSON Polje Obavezno? FHIR Resurs (iza kulisa)
anamnesisDAObservation (code=15, "Anamneza") + Composition sekcija
findingOpcionalnoComposition sekcija "Klinički nalaz i status"
recommendationOpcionalnoComposition sekcije "Terapija" + "Preporuka"
diagnosisCodeDACondition.code (icd10-hr / MKB-10)
typeDAComposition.type (011/012/013)
visitIdDAEncounter.identifier (CEZIH Visit ID)
caseIdDACondition.identifier (CEZIH Case ID)
patientMboDAPatient.identifier (MBO)

5. Infrastruktura i Šifrarnici

OID Generiranje (TC6)

CEZIH zahtijeva da svaki dokument, posjeta i slučaj ima globalno jedinstveni OID identifikator (Object Identifier). Middleware ovo radi potpuno automatski — vaša aplikacija ne treba generirati OID-ove. Svaki poziv na /api/document/send interno poziva TC6 servis.

🔧 Interno — nema API poziva.
OID-ovi se generiraju automatski pri svakom slanju dokumenta (TC18), kreiranju posjete (TC12), i kreiranju slučaja (TC16). G9 developeri ne trebaju pozivati ovaj servis direktno.

Primjer generiranog OID-a

"urn:oid:2.16.840.1.113883.3.4424.13.10.11.7.1.1748291..."

Terminologija — Pretraga Dijagnoza MKB-10 (TC7 / TC8)

Middleware automatski sinkronizira CEZIH-ove šifrarnike: TC7 preuzima CodeSystems (MKB-10, postupci, lijekovi), TC8 preuzima ValueSets (skupove vrijednosti). Vaša aplikacija može koristiti ovaj endpoint za "autocomplete" pretragu dijagnoza prilikom pisanja nalaza.

GET/api/terminology/diagnoses?q={upit}

Odgovor (Response)

{ "success": true, "results": [ { "code": "J10", "display": "Gripa (influenca) uzrokovana..." }, { "code": "J11", "display": "Gripa, virus nije utvrđen" } ] }

mCSD Registar — Liječnici i Ustanove (TC9)

TC9 omogućuje pretraživanje centralnog CEZIH mCSD registra (IHE ITI-90). Ukoliko trebate validirati OIB liječnika, šifru organizacije ili tražiti ustanove na razini države, Middleware vam daje pristup tom registru.

GET/api/registry/Practitioner (Liječnici)
GET/api/registry/Organization (Ustanove)
Vraća FHIR mCSD Bundle resurse.
